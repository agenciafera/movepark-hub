## 🔴 CRÍTICO

### S1 · Funções `api_*` executáveis por `anon`/`authenticated` (Broken Access Control) — ✅ corrigido
As 32 funções `api_*` são `SECURITY DEFINER` que **confiam no gateway**: recebem `company_id` por
parâmetro e **não checam `auth.uid()`** (o gateway resolve a empresa via chave+escopo). Elas estavam
com `EXECUTE` para **`anon`** e **`authenticated`**. Como a anon key é **pública** (vai no bundle do
front), qualquer pessoa podia chamar direto `POST /rest/v1/rpc/api_*` com **qualquer `company_id`** e:
- **escrever** catálogo/cupons/descontos/serviços/preço (`api_upsert_*`, `api_update_parking_type`, `api_delete_*`, `api_set_*`);
- **ler** reservas e ocupação de qualquer empresa (`api_list_bookings`, `api_get_booking`, `api_location_occupancy`) → vazamento de PII (risco LGPD);
- **cancelar** reservas e **criar** reservas atribuídas a qualquer empresa (`api_cancel_booking`, `api_create_booking`).

Isso **burlava todo o sistema de chaves/escopo** (a Public API inteira). O gateway chama essas
funções com **`service_role`**, então **revogar `anon`/`authenticated` não quebra nada**.
**Fix:** `20260715000000_security_hardening_scan1.sql` revoga `EXECUTE` de `anon, authenticated` em
todas as `api_*`. Validado: **0/34** `api_*` continuam anon/auth-executáveis; gateway intacto.
(`api_set_pricing`/`api_set_date_blocked` já estavam corretas — bom sinal do padrão.)

## 🟠 ALTO

### S2 · `function_search_path_mutable` em 4 funções — ✅ corrigido
`_apply_pricing` (2 overloads), `get_pricing_data`, `min_stay_satisfied` eram `SECURITY DEFINER`
**sem `search_path` fixo** → sujeitas a sequestro de resolução de nome (um schema malicioso no
`search_path` do chamador poderia mascarar uma função/tabela). **Fix:** `ALTER FUNCTION … SET
search_path = 'public','pg_temp'` (mesma migration; não toca a lógica → motor de preço inalterado).

## 🟡 MÉDIO

### S3 · Dependência client-facing com open redirect (`react-router` < 6.30.4) — 🔶 validar
`react-router-dom › react-router` (6.7.0–6.30.3): redirect same-origin com path iniciando em `//`
vira **open redirect** (GHSA-2j2x-hqr9-3h42). É a única vuln que **chega ao runtime do cliente**.
**Remediação:** `bun update react-router-dom` (fix está dentro do range `^6`); rodar o gate e
confirmar o lockfile. (Atenção ao pino de versão do bun — commitar `package.json`+`bun.lock` juntos.)

### S4 · `.env` versionado (sem `.env` no `.gitignore`) — 🔶 validar
`.env` está rastreado, hoje **só com valores públicos** (`VITE_SUPABASE_URL` + anon key) — **não é
vazamento**. Mas o `.gitignore` ignora só `.env.local`/`.env.*.local`; um `service_role`/secret
adicionado por engano ao `.env` seria commitado. **Remediação:** decidir entre (a) gitignorar `.env`
e `git rm --cached .env` (pode afetar onboarding "clone-and-run"), ou (b) manter como template
público e **garantir por política** que segredo só vai em `.env.local`/secrets. O secret-scan da
rotina tem allowlist pra anon key (ruído conhecido).

### S5 · Sem CSP (Content-Security-Policy) no deploy — 🔶 validar
`public/_headers` agora tem X-Frame-Options/X-Content-Type-Options/Referrer-Policy/HSTS/
Permissions-Policy (✅ adicionados nesta rodada), mas **falta CSP** — a defesa mais forte contra XSS.
CSP exige allowlist cuidadosa dos domínios usados (Supabase `*.supabase.co`, Pagar.me
`api.pagar.me`/`checkout`, OpenStreetMap/tiles, Google OAuth/Fonts, próprio `hub.movepark.co`/
`api.movepark.co`). **Remediação:** montar CSP `report-only` primeiro, observar violações, então
endurecer. Tarefa própria.

### S6 · `rls_enabled_no_policy` em `payment_webhook_event` — 🔶 validar
RLS habilitada mas **sem policy** → nega tudo para `anon`/`authenticated` (seguro por padrão). A
tabela é escrita só por `service_role` (idempotência de webhook), que bypassa RLS. Provavelmente
**intencional**; confirmar que nenhuma leitura via cliente é esperada e, se quiser, documentar com
uma policy explícita "deny" ou um comentário.

## ⚪ BAIXO / INFO

### S7 · `extension_in_public` — `pg_net` no schema `public` — 🔶 validar
Recomendação Supabase: extensões fora do `public`. Mover `pg_net` exige cuidado (os crons
`wl-deliver`/`wl-reconcile`/`wps-deliver` usam `net.http_post`). Baixo risco; reagendar com schema
dedicado num momento de manutenção.

### S8 · Vulns de toolchain (dev/build only) — ⏭️ aceito / atualizar quando conveniente
`undici`/`form-data`/`ws`/`esbuild`/`js-yaml` vêm de **wrangler, vite-react-ssg(jsdom), vitest,
eslint** — **não vão pro bundle do cliente nem pro runtime de produção**. Risco real baixo. Resolver
com `bun update` na próxima janela (junto do S3).

### S9 · `auth_leaked_password_protection` desabilitado — 🔶 validar
Proteção de senha vazada (HaveIBeenPwned) **desligada** no Supabase Auth. Como o backoffice usa
senha, **ligar** em Dashboard → Authentication → Policies (1 clique). Baixo esforço, bom ganho.
